# 병원 내부 직원용 통합 안전 관리 앱 — PRD

## 1. 한눈에 보기 (Overview)

**한 줄 컨셉**
> 흩어진 환자 정보를 한 화면에 모으고, 위험한 순간엔 먼저 알려줘서 실수를 막는 **병원 내부 직원용 통합 안전 관리 앱**.

| 항목 | 내용 |
|---|---|
| 대상 사용자 | 중간 규모 병원의 내부 직원 (의사·간호사·원무과 등 모두) |
| 사용 환경 | 휴대폰 앱 + 컴퓨터(웹) 둘 다 |
| 프로젝트 성격 | 연습/취미 (바이브코딩 학습용) |
| 핵심 가치 | 정보 통합 · 환자 안전 · 부서 간 매끄러운 협업 |

---

## 2. 풀려는 문제 (Problem)

병원에서 한 환자의 정보(접수 기록, 진료 차트, 검사 결과, 투약 내역 등)가 **여러 프로그램에 흩어져 있다.** 직원들은 프로그램 사이를 오가며 정보를 찾느라 시간을 쓰고, 그 과정에서 중요한 것을 **놓치는 위험**이 생긴다.

- 환자 한 명을 파악하려면 여러 화면을 열어야 함
- 부서 간 전달이 전화·메모·구두라 누락·지연 발생
- 알레르기, 투약 시간, 필수 절차 같은 **안전 정보**를 놓치면 사고로 이어질 수 있음

---

## 3. 목표와 성공 기준 (Goals & Success)

**목표**
1. 환자 한 명의 핵심 정보를 **한 화면**에서 본다.
2. 위험한 순간(알레르기·투약·빠진 절차)을 **미리 알림**으로 막는다.
3. 부서 간 인수인계를 **자동으로** 흐르게 한다.

**성공 기준 (취미 프로젝트 수준)** [추측]
- 한 환자의 정보를 보기 위해 열어야 하는 화면이 **1개**로 줄어든다.
- 위험 알림이 뜨면 직원이 조치하기 전까지 화면에 계속 남는다.
- 한 부서에서 입력한 내용이 **다시 입력 없이** 다음 부서 화면에 나타난다.

**반대로 경계할 점 (Counter-metric)**
- 알림이 너무 많아 직원이 무시하게 되는 "알림 피로"를 만들지 않는다.

---

## 4. 핵심 기능 (Features & Requirements)

> FR = 기능 요구사항(Functional Requirement). 각 기능에 번호를 붙여 나중에 추적하기 쉽게 함.

### 기능 그룹 A — 환자 정보 통합 (기반)
모든 기능의 바탕이 되는 부분. 먼저 만들어야 함.

- **FR1.** 환자 한 명을 선택하면, 기본 정보·방문 기록·진단·투약·검사 결과·수납 상태를 **한 화면**에서 볼 수 있다.
- **FR2.** 환자를 이름 또는 등록번호로 빠르게 검색할 수 있다.
- **FR3.** 직원이 새 정보(진료 기록, 검사 결과 등)를 입력하면 해당 환자 화면에 즉시 반영된다.

### 기능 그룹 B — 환자 안전 알림 (🥇 1순위)
- **FR4.** 환자에게 등록된 **알레르기/금기 약물**과 충돌하는 처방을 입력하면 **경고 알림**을 띄운다.
- **FR5.** 정해진 **투약 시간**이 되면 담당 직원에게 알림을 보낸다.
- **FR6.** 시술·수술 등 주요 절차에 대해 **필수 체크리스트**(예: 동의서, 금식 확인)를 제공하고, 다 체크해야 다음 단계로 넘어갈 수 있다.
- **FR7.** 안전 알림은 직원이 **확인·조치**하기 전까지 화면에서 사라지지 않는다.

### 기능 그룹 C — 부서 간 자동 인수인계 (🥈 2순위)
- **FR8.** 한 부서(예: 간호)가 입력·완료한 내용은 다음 부서(예: 진료, 약국) 화면에 **자동으로 전달**된다.

### 기능 그룹 D — 대기시간·여정 추적 (🥉 3순위)
- **FR9.** 환자가 지금 어느 단계(접수 → 진료 → 검사 → 수납)에 있는지 **실시간으로 표시**한다(배달앱처럼).
- **FR10.** 한 환자가 한 단계에서 너무 오래 머무르면(대기 초과) 담당 직원에게 알림을 보낸다.

---

## 5. 비기능 요구사항 (NFR — 공통 사항)

> 특정 기능이 아니라 앱 전체에 걸친 조건.

- **NFR1. 멀티 화면 지원** — 휴대폰과 컴퓨터(웹) 양쪽에서 모두 잘 보이고 동작해야 한다.
- **NFR2. 단순 접근(1차)** — 로그인한 직원이면 환자 정보를 볼 수 있다. 직군별 세밀한 권한 구분은 1차 범위 밖(나중에 추가).
- **NFR3. 정보 보호** — 환자 정보는 민감하므로 로그인한 직원만 접근할 수 있어야 한다. (취미 단계에서는 간단한 로그인 수준)
- **NFR4. 빠른 반응** — 한 부서의 입력이 다른 부서 화면에 거의 즉시(몇 초 내) 보여야 한다.
- **NFR5. 쉬운 사용** — 바쁜 의료 현장 특성상 화면이 직관적이고 클릭 수가 적어야 한다.

---

## 6. 이번 범위 밖 (Out of Scope)

취미·학습용 1차 버전에서는 **다루지 않는 것**들. (나중에 키울 수 있는 씨앗)

- 건강 변화 그래프(혈압·혈당 추이 시각화)
- 위험 처치 이중 확인(2명 승인) 자동화
- **놓침 방지 에스컬레이션** — 일정 시간 조치가 없으면 상급자에게 자동 보고 (1차에선 제외, 다음 버전 후보)
- 직군별 세밀한 권한 구분
- 병원 전체 통계 대시보드(어느 과가 붐비는지 등)
- 실제 의료기기·외부 시스템 연동, 법적 인증(의료기기 인증 등)

---

## 7. 가정 및 열린 질문 (Assumptions & Open Questions)

> 나중에 확인하거나 결정하면 좋은 것들.

- [확정] 권한은 1차에서 단순하게(로그인 직원이면 열람 가능). 세밀한 권한은 다음 버전.
- [확정] 에스컬레이션(상급자 자동 보고)은 1차 제외, 다음 버전 후보.
- [추측] 환자 정보는 앱 안에 직접 입력하는 것을 전제로 함 (기존 병원 시스템과 연동은 범위 밖).
- [열린 질문] 4가지 기능 그룹 중 **가장 먼저 만들고 싶은 것**은? (보통 A → B 순서를 추천)
- [열린 질문] "대기 초과"의 기준 시간(예: 30분)을 어떻게 정할지.
