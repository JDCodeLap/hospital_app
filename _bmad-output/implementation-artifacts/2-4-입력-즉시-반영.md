# Story 2.4: 입력 즉시 반영 (실시간 자동 갱신)

Status: done

## Story

As a 병원 직원,
I want 누군가 환자 정보를 추가/수정하면 내가 보고 있는 환자 화면이 새로고침 없이 바로 갱신되기를,
so that 항상 최신 정보를 본다. (FR3 / NFR4)

## Acceptance Criteria

epics.md 원문 + 아키텍처(FastAPI WebSocket)·2.3(통합 화면)·1.3/1.4(토큰 인증)에서 도출한 검증 가능한 항목입니다.

1. **(원문)** 환자 통합 화면(`/patients/[id]`)을 보고 있을 때, 누군가 그 환자의 정보를 추가/수정하면 **화면이 새로고침 없이 몇 초 내 자동 갱신**된다. (NFR4, **FastAPI WebSocket** 사용 — 폴링 금지)
2. 백엔드에 **환자별 WebSocket 엔드포인트** `WS /ws/patients/{patient_id}` 추가 — 연결 시 **토큰 검증**(브라우저 WS는 헤더 불가 → `?token=` 쿼리로 받음). 토큰 없음/무효면 연결 거부(close). 로그인한 직원만 구독 가능(NFR3 일관).
3. 해당 환자 데이터에 **쓰기(추가/수정)가 일어나면**, 백엔드가 그 환자 방을 구독 중인 모든 WebSocket에 **갱신 신호**(`{"type":"patient_updated","patient_id":id}`)를 브로드캐스트한다.
4. **실시간을 증명할 최소 "입력" 1개**로 **방문기록 추가** 엔드포인트 `POST /api/patients/{patient_id}/visits`(보호됨)를 추가한다 — 방문 1건 저장 후 위 브로드캐스트를 호출. (정식 입력 폼 UI·다른 항목 편집은 범위 밖 — 이 엔드포인트는 `/docs`(Swagger)나 두 번째 클라이언트로 시연/검증)
5. 프론트 `PatientDetail`(2.3)이 해당 환자의 WS를 **구독**하고, `patient_updated` 신호를 받으면 **기존 `GET /api/patients/{id}` 재호출(reload)** 로 화면을 갱신한다(전체 새로고침 없이).
6. **연결 수명주기**: 화면 진입/`id` 변경 시 연결 열고, 화면 이탈/`id` 변경 시 닫는다(누수 방지). 연결이 끊기면 **간단한 재연결**(몇 초 후 1~2회 시도). **WS가 안 되어도 화면은 정상 동작**(2.3 초기 fetch로 이미 표시됨) — WS 실패는 사용자에게 오류로 표시하지 않음(조용히 실패).
7. **2.3 화면 동작을 깨지 않는다** — 기존 로딩/404/401/오류 상태, 카드 레이아웃, AuthGuard 보호, 클라이언트 fetch(1.4 가이드) 모두 유지. WS는 그 위에 얹는다.

> **범위 밖(하지 말 것):** 정식 입력/편집 폼 UI 및 모든 항목 CRUD(후속) — 이 스토리는 **실시간 반영 파이프**가 목적이고 쓰기는 시연용 방문기록 1종만. / 역할별 권한·WS Origin 화이트리스트·낙관적 업데이트(optimistic UI)·타이핑/접속자 표시(presence) = 범위 밖. / Epic 3·4의 개별 쓰기(투약 완료·단계 변경 등)는 후속에서 같은 broadcast 헬퍼를 재사용.

## Dev Notes

### 이 스토리의 핵심 (한 줄)
2.3 통합 화면에 **실시간 파이프(WebSocket)** 를 얹는다: 백엔드가 환자별 WS 방으로 "이 환자 바뀜" 신호를 쏘면, 그 환자 화면을 보던 클라이언트가 **새로고침 없이** `GET /api/patients/{id}`를 다시 불러 갱신한다. 변화를 일으킬 최소 입력으로 **방문기록 추가** 엔드포인트 1개를 둔다.

### ⚠️ 이 스토리의 가장 중요한 설계 판단
- **읽기 전용 앱에 '입력'이 없음 → 실시간을 증명할 최소 쓰기 1개 필요.** 그래서 `POST /api/patients/{id}/visits`를 둔다(방문기록 추가). 정식 입력 폼 UI나 다른 항목 편집은 **이 스토리 범위가 아님** — 시연/검증은 `/docs`(Swagger)나 두 번째 클라이언트로 한다. (Epic 3·4의 실제 쓰기가 생기면 같은 `manager.broadcast`를 호출해 재사용)
- **브라우저 WebSocket은 커스텀 헤더(Authorization) 불가** → 토큰을 `?token=` 쿼리로 전달해 서버에서 검증. (토큰이 URL/서버로그에 남는 점은 보안 deferred로 기록 — 운영 시 단명 토큰/쿠키 기반 등 검토)
- **WS는 보조 채널.** 진짜 데이터는 항상 `GET /api/patients/{id}`(2.3, 보호됨)에서 온다. WS는 "다시 불러와" 신호만 보냄(데이터 자체를 WS로 보내지 않음) → 권한·검증을 읽기 API가 일관되게 담당, 구현 단순.

### 확정 기술/패턴 가이드

- **백엔드 ConnectionManager(요지):**
  ```python
  # realtime.py
  from fastapi import WebSocket
  class ConnectionManager:
      def __init__(self) -> None:
          self.rooms: dict[int, set[WebSocket]] = {}
      async def connect(self, patient_id: int, ws: WebSocket) -> None:
          await ws.accept()
          self.rooms.setdefault(patient_id, set()).add(ws)
      def disconnect(self, patient_id: int, ws: WebSocket) -> None:
          room = self.rooms.get(patient_id)
          if room:
              room.discard(ws)
              if not room:
                  self.rooms.pop(patient_id, None)
      async def broadcast(self, patient_id: int, message: dict) -> None:
          for ws in list(self.rooms.get(patient_id, set())):
              try:
                  await ws.send_json(message)
              except Exception:
                  self.disconnect(patient_id, ws)  # 죽은 연결 정리
  manager = ConnectionManager()
  ```
- **WS 엔드포인트(요지):**
  ```python
  from fastapi import WebSocket, WebSocketDisconnect
  from .realtime import manager
  @app.websocket("/ws/patients/{patient_id}")
  async def patient_ws(websocket: WebSocket, patient_id: int):
      token = websocket.query_params.get("token")
      with Session(engine) as session:
          staff = authenticate_ws_token(token, session)
      if staff is None:
          await websocket.close(code=1008)  # 정책 위반(미인증)
          return
      await manager.connect(patient_id, websocket)
      try:
          while True:
              await websocket.receive_text()  # 클라 메시지는 무시(연결 유지용)
      except WebSocketDisconnect:
          manager.disconnect(patient_id, websocket)
  ```
  - `engine`은 `from .database import engine`. (WS는 요청 스코프 세션 의존성 대신 짧게 Session 직접 사용)
- **시연 쓰기(요지):**
  ```python
  from datetime import datetime
  from pydantic import BaseModel
  class VisitIn(BaseModel):
      department: str
      reason: str = ""
  @app.post("/api/patients/{patient_id}/visits")
  async def add_visit(patient_id: int, payload: VisitIn,
                      current: Staff = Depends(get_current_user),
                      session: Session = Depends(get_session)) -> dict:
      if session.get(Patient, patient_id) is None:
          raise HTTPException(status_code=404, detail="환자를 찾을 수 없습니다")
      visit = Visit(patient_id=patient_id, visited_at=datetime.now(),
                    department=payload.department, reason=payload.reason)
      session.add(visit); session.commit(); session.refresh(visit)
      await manager.broadcast(patient_id, {"type": "patient_updated", "patient_id": patient_id})
      return {"id": visit.id, "patient_id": patient_id}
  ```
- **프론트 WS 구독(요지):**
  ```tsx
  // lib/api.ts
  export const WS_BASE = API_BASE.replace(/^http/, "ws"); // http→ws, https→wss

  // PatientDetail.tsx — 별도 useEffect([id])
  useEffect(() => {
    const token = getToken();
    if (!id || !token) return;
    let closedByUs = false;
    let ws: WebSocket | null = null;
    let retry = 0;
    const open = () => {
      ws = new WebSocket(`${WS_BASE}/ws/patients/${id}?token=${encodeURIComponent(token)}`);
      ws.onmessage = (e) => {
        try { const m = JSON.parse(e.data); if (m.type === "patient_updated") reload(); } catch {}
      };
      ws.onclose = () => { if (!closedByUs && retry < 2) { retry++; setTimeout(open, 3000); } };
    };
    open();
    return () => { closedByUs = true; ws?.close(); };
  }, [id]);
  ```
  - `reload()` = 2.3의 bundle fetch를 다시 부르는 함수(상태분기 재사용). WS 실패/끊김은 화면 오류로 표시하지 않음(조용히).

### 흔한 실수 방지
- ❌ WS 토큰을 헤더로 받으려 함 → 브라우저 WS는 헤더 불가. **쿼리 `?token=`** 사용.
- ❌ WS로 환자 데이터 자체를 내려보냄 → 권한/검증 일관성 위해 **신호만** 보내고 데이터는 `GET /api/patients/{id}` 재호출로.
- ❌ WS 실패를 화면 오류로 표시 → 보조 채널이므로 조용히 실패(초기 fetch로 화면은 이미 정상).
- ❌ 폴링(setInterval로 반복 GET)으로 구현 → NFR4는 **WebSocket** 명시. 폴링 금지.
- ❌ 연결 정리 누락(언마운트/ id 변경 시 close 안 함) → 연결 누수·중복 갱신. cleanup에서 반드시 close.
