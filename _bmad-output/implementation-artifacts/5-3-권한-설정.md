# Story 5.3: 권한 설정

Status: ready-for-dev

## Story

As a 병원 관리자,
I want 직원별로 **볼 수 있는 정보 영역(방문·진단·투약·검사·수납)**을 설정하기를,
So that 각 직원이 자기 일에 필요한 만큼만 접근하게 한다. (FR12)

## Acceptance Criteria

1. **접근 범위 설정(관리자)** — 관리자가 직원 등록/수정 폼에서 그 직원이 볼 수 있는 **정보 영역**을 설정한다. 영역은 **방문·진단·투약·검사·수납 5종**이며, **"전체 접근"** 또는 개별 선택이 가능하다. 저장하면 즉시 반영된다.
2. **범위 저장·표시·기본값** — 직원 목록에 각 직원의 접근 범위가 배지/요약("전체" 또는 영역 목록)으로 보인다. **기본값은 "전체"** — 기존 직원과 신규 직원은 지금까지처럼 모든 영역을 본다(하위호환, 회귀 없음).
3. **읽기 강제(백엔드가 진짜 자물쇠)** — 일반 직원이 환자 통합화면을 열면 `GET /api/patients/{id}` 응답에 **허용된 영역만 포함**되고, 범위 밖 영역은 **응답에서 제외**된다(프론트 숨김이 아니라 백엔드가 데이터를 안 줌 — 직접 API 호출/화면 조작으로도 못 봄, NFR3). 응답에 `visible_sections`(허용 영역 키 목록)를 함께 내려준다.
4. **항상 보이는 안전·협업 기본** — **기본정보·알레르기/안전경고(safety_ack)·필수절차 체크리스트(checklist)·인계메모(handover_notes)·단계 타임라인**은 접근 범위와 무관하게 **항상 보인다**. (환자 안전·부서 협업은 누구도 놓치면 안 되므로 게이트하지 않는다.)
5. **쓰기 강제(백엔드)** — 범위 밖 영역에 대한 **쓰기도 막는다**: **투약** 추가/수정/삭제는 "투약" 영역이 없으면 **403**, **방문** 추가는 "방문" 영역이 없으면 **403**. 미인증은 **401**(403=권한 없음과 구분 — 2-3 deferred 이행).
6. **관리자는 전체 + 정규화** — `role=="admin"`인 직원은 `access_scope`와 무관하게 **항상 모든 영역**을 본다(관리자 전권). 저장 시점에 `access_scope`를 정규화한다: **`"all"` 또는 5개 유효 영역 키의 부분집합만** 허용(그 외 값은 422). 5개를 다 고르면 `"all"`로 단순화한다.
7. **관리자 전용 + 보호** — 접근 범위 설정(직원 생성/수정 API)은 **관리자만**(`get_current_admin` 재사용, NFR2). 일반 직원이 호출하면 **403**, 미인증은 **401**.
8. **프론트 표시(범위 밖 구분)** — 통합화면에서 범위 밖 카드는 **"이 정보는 접근 권한이 없습니다"**(자물쇠 아이콘+글자, UX-DR8) 안내로 대체한다 — 데이터가 없어 "기록 없음"인 것과 **명확히 구분**. 범위 밖이면 그 영역의 **입력 폼(처방 폼 등)도 숨긴다**. 범위 밖 쓰기 시 받은 **403은 강제 로그아웃이 아니라 인라인 "권한 없음" 안내**로 처리한다.

> **범위 밖(하지 말 것):**
> - **어떤 "환자"를 보는지(행 단위 필터)** — 5.3은 정보 "영역(섹션)" 가시성만 다룬다. 부서/담당 환자 배정 기반 환자 목록 제한은 후속(현재 Patient엔 담당 부서/직원 필드가 없음 → 모델 확장 필요). 모든 직원은 지금처럼 모든 **환자**를 검색·열람하되, **영역만** 제한된다.
> - **알림·흐름판·대시보드 영역 게이팅** — `/api/medication-alerts`·`administer`·`/board`·`/api/patients`(목록 요약)는 이번에 게이트하지 않는다(후속). 5.3은 **환자 통합화면(`GET /api/patients/{id}`)의 영역 가시성 + 투약/방문 쓰기**에 집중.
> - **safety_ack·checklist·handover·advance-stage 쓰기 게이팅** — 전 직원 공통(안전·협업). 게이트 안 함.
> - **접근 감사 로그**(누가 무엇을 봤는지) — 후속.
> - **전체 현황 대시보드 = Story 5.4(FR13)** / **기준값 설정 = Story 5.5(FR14)**.

  - [ ] `normalize_scope(raw)` — 쓰기 시점 정규화(`normalize_role`과 같은 위치/철학):
    ```python
    def normalize_scope(raw: str | None) -> str:
        """'all' 또는 유효 영역 키 콤마 목록만 허용. 빈값/None/'all'→'all'.
        무효 키 있으면 422. 5개 다 고르면 'all'로 단순화(저장값 표준화)."""
        if raw is None:
            return "all"
        s = raw.strip().lower()
        if s in ("", "all"):
            return "all"
        keys = [k.strip() for k in s.split(",") if k.strip()]
        bad = [k for k in keys if k not in ALL_SECTION_KEYS]
        if bad:
            raise HTTPException(status_code=422,
                detail=f"알 수 없는 접근 영역: {', '.join(bad)}")
        uniq = sorted(set(keys))
        return "all" if set(uniq) == ALL_SECTION_KEYS else ",".join(uniq)
    ```
  - [ ] `allowed_sections(staff)` — 읽기/쓰기 판정 공통:
    ```python
    def allowed_sections(staff: Staff) -> set[str]:
        """이 직원이 볼 수 있는 영역 키 집합. 관리자는 전체. 'all'/빈값도 전체."""
        if staff.role == "admin":
            return set(ALL_SECTION_KEYS)
        scope = (staff.access_scope or "").strip().lower()
        if scope in ("", "all"):
            return set(ALL_SECTION_KEYS)
        return {k for k in (x.strip() for x in scope.split(",")) if k in ALL_SECTION_KEYS}
    ```
  - [ ] `require_section(staff, key)` — 쓰기 보호:
    ```python
    def require_section(staff: Staff, section_key: str) -> None:
        """쓰기 보호: 해당 영역 권한이 없으면 403(401=미인증과 구분)."""
        if section_key not in allowed_sections(staff):
            raise HTTPException(status_code=403, detail="이 정보 영역에 접근 권한이 없습니다")
    ```

## Dev Notes

### 이 스토리의 핵심 (한 줄)
**관리자가 직원별 "정보 영역 접근 범위"를 정하고, 백엔드가 그걸 강제한다.** `Staff.access_scope`(5.2 `job_title`처럼 멱등 컬럼 추가) + `get_patient`가 허용 영역만 반환 + 투약/방문 쓰기 403 게이트 + 5.2 직원 폼에 범위 편집 UI + 통합화면이 범위 밖 카드를 "권한 없음"으로 표시.

### ⚠️ 이 스토리의 가장 중요한 설계 판단

1. **"정보 범위" = 영역(섹션) 가시성 (행 단위 환자 필터 아님).**
   - FR12 "볼 수 있는 정보 범위" / 에픽 AC "설정된 범위만 보인다"를, 기존 통합화면이 이미 가진 섹션(방문·진단·투약·검사·수납)에 매핑해 구현한다 — 가장 직접적이고 데모 가능.
   - **어떤 "환자"를 보는지**(행 필터)는 범위 밖: Patient에 담당 부서/직원 필드가 없어 모델 확장이 필요(후속). 모든 직원은 모든 환자를 보되 **영역만** 제한.

2. **3축 분리: `role`(접근 게이트) · `job_title`(분류) · `access_scope`(정보 영역).** (5.1→5.2 인계의 연장)
   - `role`=admin/staff: 관리자 페이지 접근(`get_current_admin`). **세분화 아님.**
   - `job_title`=의사/간호사/원무과/기타: 분류·표시(화면 제한 없음).
   - `access_scope`=영역 키 집합: **이 스토리가 추가하는 "정보 영역" 축.** role과 섞지 말 것(관리자는 scope 무관 전체).

3. **진짜 자물쇠는 백엔드.** (3.1/5.1/5.2 철학 일관) — 프론트 카드 숨김·폼 숨김은 UX 보조. **`get_patient`가 범위 밖 데이터를 응답에 아예 안 담고**, 투약/방문 쓰기는 `require_section`이 **403**. 화면/토큰을 조작해도 못 보고 못 쓴다(NFR3).

4. **기본값 'all' = 무회귀.** 기존 직원·신규 직원·관리자는 전부 전체 영역을 본다 → 5.3 도입으로 **기존 동작이 하나도 안 바뀐다**. 관리자가 명시적으로 좁힐 때만 제한 발생.

5. **안전·협업은 게이트하지 않는다.** 알레르기/안전경고(safety_ack)·필수절차(checklist)·인계메모(handover)·기본정보·단계는 **누구나 항상** 본다 — 환자 안전·부서 협업은 권한으로 가리면 위험. (3.2/3.4/4.2 취지.)

6. **403(권한 없음) ≠ 401(미인증).** 쓰기 게이트는 403을 쓰고, 프론트는 403을 **강제 로그아웃이 아니라 "권한 없음" 안내**로 처리 → **2-3 deferred("403→강제 로그아웃 재검토") 이행.** (정상 UI에선 범위 밖 폼을 안 그리므로 거의 발생 안 하지만, 직접 호출 방어 + 정책 정립.)

7. **`visible_sections`를 응답에 함께 내려 프론트가 "권한 없음"과 "기록 없음"을 구분.** 단순히 키를 빼면 프론트가 `?? []`로 "기록 없음"처럼 보일 수 있음 → 명시적 허용 목록으로 카드 자체를 "권한 없음"으로 대체(AC8).

### 흔한 실수 방지

- ❌ `access_scope`를 `role`/`job_title`에 섞기 → 3축 혼란·게이트 붕괴. **별도 컬럼.**
- ❌ 프론트 카드 숨김만 하고 백엔드는 다 반환 → 직접 호출로 노출(NFR3 위반). **`get_patient`가 키를 빼야 진짜 자물쇠.**
- ❌ 응답 키 이름 바꾸기(`lab_results`→`labs`) → 프론트·하위호환 깨짐. **기존 키 그대로**, scope key(`labs`/`billing`)와 bundle key(`lab_results`/`billings`) 매핑만.
- ❌ 관리자에게도 scope 적용 → 관리자 잠금. **admin은 항상 전체**(`allowed_sections` 첫 줄).
- ❌ 기본값을 빈/none으로 두기 → 기존 직원이 아무것도 못 봄(대형 회귀). **기본 'all'**.
- ❌ 안전경고/체크리스트/인계메모 게이트 → 안전·협업 정보 가림(위험). **항상 표시.**
- ❌ 범위 밖 쓰기를 401로 → 강제 로그아웃 오작동. **403 + 프론트 인라인 안내**(2-3 deferred).
- ❌ 프론트가 키 없음을 `?? []`로 "기록 없음" 표시 → "권한 없음"과 혼동. **`visible_sections`로 분기.**
- ❌ `normalize_scope` 빠뜨려 `"의사,billing"` 같은 값 저장 → 판정 불일치. **쓰기 시 정규화(422).**
- ❌ 무효 영역 키 무시하고 저장 → 조용한 권한 오류. **무효 키는 422로 거부.**
