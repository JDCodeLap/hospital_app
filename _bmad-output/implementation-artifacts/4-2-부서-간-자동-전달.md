# Story 4.2: 부서 간 자동 전달

Status: done

## Story

As a 병원 직원,
I want 이전 부서가 남긴 인계 메모를 환자 화면에서 바로 보고, 내 부서의 인계 메모도 남길 수 있기를,
So that 전화·메모지 없이도 이전 부서 처리 내용을 파악하고 다음 부서에 전달할 수 있다. (FR8)

## Acceptance Criteria

1. 환자 통합 화면(`/patients/{id}`)에 **"인계 메모"** 섹션이 있다 — 이 환자에 대해 지금까지 남긴 메모들이 최신순으로 표시된다(없으면 "기록 없음").
2. 각 메모는 **메모 내용 + 남긴 시각 + 남긴 단계 + 작성자 이름**을 표시한다(누가·언제·어느 단계에서 남겼는지 파악).
3. **"인계 메모 남기기"** 폼이 섹션 안(또는 아래)에 있다 — textarea + 제출 버튼(최소 44px). 빈 메모는 제출 불가.
4. 메모를 저장하면 **같은 환자 화면을 보고 있는 모든 클라이언트**에 즉시 반영된다(2.4의 `manager.broadcast` 재사용 — WebSocket 실시간).
5. 저장 성공 시 textarea가 초기화되고, 페이지 새로고침 없이 새 메모가 목록 최상단에 나타난다(실시간 갱신).
6. **로그인한 직원만** 메모를 조회·작성할 수 있다(NFR3 — `get_current_user` 보호).
7. **백엔드 `GET /api/patients/{id}` 묶음**에 `handover_notes` 키가 추가된다 — 최신 5건, 기존 키(`patient`, `visits`, `diagnoses`, `medications`, `lab_results`, `billings`, `safety_ack`, `checklist`) 전부 유지(하위호환).
8. 기존 배너(SafetyBanner)·타임라인(StageTimeline)·체크리스트(ProcedureChecklist)·처방폼(PrescribeForm)·카드들·WebSocket 동작을 전혀 깨지 않는다.

> **범위 밖(하지 말 것):** 메모 삭제/수정 = 후속 / 단계별 메모 필터(현재 단계만 표시) = 후속 / 담당자 지정(특정 부서 직원에게만 표시) = Epic 5 권한 설정과 함께 / 대기 초과 알림(4.4) / 환자 흐름판(4.3) / 메모 알림 배지

## Dev Notes

### 이 스토리의 핵심 (한 줄)
새 `HandoverNote` 테이블 + `POST /handover-note` 엔드포인트 + GET 묶음에 추가 + `HandoverNoteSection` 컴포넌트로 **인계 메모 작성·표시** 기능을 구현한다. 실시간은 2.4 broadcast 재사용으로 공짜.

### ⚠️ 이 스토리의 가장 중요한 설계 판단

1. **`from_stage`는 서버가 자동 기록** — 클라이언트가 보내는 것이 아니라, 서버가 `patient.current_stage` 값을 저장 시점에 기록. 프론트가 조작할 수 없고, 작성 시점의 단계가 정확히 기록됨.

2. **메모는 append-only** — 삭제/수정 없음. 감사 추적 목적으로 남겨둔다(범위 밖).

3. **`create_all`로 새 테이블 생성** — `HandoverNote`는 기존 테이블과 무관한 새 테이블. `init_db()`의 `SQLModel.metadata.create_all(engine)`이 서버 시작 시 자동 생성. **Alembic 마이그레이션 불필요**(3.2 SafetyAck, 3.3 MedicationAdministration, 3.4 ChecklistCheck와 동일한 패턴).

4. **author_id → full_name 조인은 GET에서만** — POST 응답에는 id만 반환하고, 이름은 GET 묶음에서 `Staff` 테이블을 n+1 조회로 조인. 노트가 최대 5건이라 성능 문제 없음.

5. **실시간은 공짜** — `POST /handover-note` 성공 후 `manager.broadcast(patient_id, ...)` 한 줄이면 같은 환자 화면을 보는 모든 클라이언트의 `ws.onmessage`가 `loadBundle(undefined, true)` 호출 → 메모 목록 자동 갱신. 별도 WS 채널 불필요.

6. **빈 메모 422** — `payload.note.strip()` 후 비면 422 반환. 프론트도 빈 textarea면 버튼 비활성이지만, 직접 API 호출 방어.

### 확정 기술/패턴 가이드

```python
# backend/app/main.py 에 추가

class HandoverNoteIn(BaseModel):
    """인계 메모 작성 요청 본문(Story 4.2, FR8)."""
    note: str

@app.post("/api/patients/{patient_id}/handover-note")
async def add_handover_note(
    patient_id: int,
    payload: HandoverNoteIn,
    current: Staff = Depends(get_current_user),
    session: Session = Depends(get_session),
) -> dict[str, object]:
    patient = session.get(Patient, patient_id)
    if patient is None:
        raise HTTPException(status_code=404, detail="환자를 찾을 수 없습니다")
    note_text = payload.note.strip()
    if not note_text:
        raise HTTPException(status_code=422, detail="메모 내용을 입력하세요")
    note = HandoverNote(
        patient_id=patient_id,
        from_stage=patient.current_stage,
        note=note_text,
        author_id=current.id,
        created_at=datetime.now(),
    )
    session.add(note)
    session.commit()
    session.refresh(note)
    await manager.broadcast(
        patient_id, {"type": "patient_updated", "patient_id": patient_id}
    )
    return {"id": note.id, "patient_id": patient_id}
```

- ❌ `HandoverNote` 추가하면서 `init_db()` 또는 `create_all` 호출 빠뜨리기 → `lifespan`의 `init_db()` 안에 이미 `create_all`이 있음. 아무것도 추가할 필요 없음.
- ❌ GET 묶음에 `handover_notes` 추가하면서 기존 키(`patient`, `visits`, `checklist` 등) 실수로 제거 → **기존 키 전부 유지**. 새 키를 맨 뒤에 추가.
- ❌ `author_id`만 반환하고 이름 없이 → GET에서 `Staff.full_name`을 join해 `author_name`으로 포함해야 AC #2 충족.
- ❌ `submittingRef` 없이 textarea + 버튼 구현 → 느린 네트워크에서 이중 저장 발생. 3.1 PrescribeForm의 `submittingRef = useRef(false)` 패턴 그대로 복사.
- ❌ `HandoverNoteSection`에 `router` 없이 401 처리 → `useRouter` 가져와서 `clearToken() + router.replace('/login')`.
- ❌ `format.ts` 없이 `HandoverNoteSection`에서 날짜 포맷 직접 구현 → `PatientDetail`의 `fmtDateTime`과 동일 함수여야 일관됨.
