# Story 1.4: 로그인 보호 & 로그아웃

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 병원 직원,
I want 로그인하지 않으면 환자 정보(보호된 화면)에 못 들어가고, 로그아웃도 할 수 있기를,
so that 환자 정보가 아무에게나 노출되지 않는다. (NFR3)

## Acceptance Criteria

epics.md 원문 + architecture(토큰 인증)·1.3에서 만든 자산(`get_current_user`, `lib/auth.ts`)에서 도출한 검증 가능한 항목입니다.

1. **(원문)** 로그인하지 않은 상태에서 보호된 화면(현재는 홈 `/`) 주소로 접근하면 **로그인 화면(`/login`)으로 돌려보낸다.**
2. **(원문)** 로그인 상태에서 **로그아웃하면 다시 로그인 화면(`/login`)으로** 간다(보관한 토큰 삭제).
3. **유효한 토큰이 있는 직원만** 보호된 화면을 본다 — 화면 진입 시 보관된 토큰으로 백엔드 `GET /api/auth/me`를 호출해 **확인되면(200) 화면 표시, 실패(401·토큰 없음/만료/위조)면 토큰 삭제 후 `/login`으로** 보낸다.
4. 보호된 화면은 **확인이 끝나기 전까지 내용(환자 정보 등)을 잠깐이라도 노출하지 않는다** — 확인 중에는 "확인 중…" 같은 빈/로딩 상태를 보여주고, 통과한 뒤에만 본문을 그린다.
5. 로그아웃 버튼은 **로그인한(보호된) 화면에서 보이고**, 누르면 토큰을 지운 뒤 `/login`으로 이동한다. 로그인 화면에는 로그아웃 버튼이 없다.
6. 로그인 화면(`/login`)은 **로그인 없이도 접근 가능**해야 한다(보호 대상 아님) — 보호 로직이 로그인 화면 자체를 막아 무한 반복(로그인↔리다이렉트)되면 안 된다.
7. 화면/버튼은 DESIGN 다크 모드 토큰을 따르고, 접근성(터치 영역 44px 이상=`h-11`, 색만으로 정보전달 금지)을 지킨다.

> **범위 밖(하지 말 것):** 역할(의사/간호사/관리자)별 권한 분기·관리자 전용 화면은 **Epic 5**. 토큰을 httpOnly 쿠키로 바꾸는 것·미들웨어(서버) 기반 보호는 **deferred**(아래 Dev Notes 참고). 이 스토리는 "미로그인 차단 + 로그아웃"까지만.

## Dev Notes

### 이 스토리의 핵심 (한 줄)
로그인 안 했거나 신분증(토큰)이 가짜/만료면 **보호된 화면을 못 보게 막아 로그인 화면으로 돌려보내고**, 로그인한 사람은 **로그아웃 버튼으로 나갈 수 있게** 한다. 백엔드는 이미 `get_current_user`로 API를 지키고 있으니, 이 스토리는 주로 **프론트(화면) 작업**이다.

### ⚠️ 가장 중요한 설계 판단 — 왜 "미들웨어"가 아니라 "화면 쪽(클라이언트) 문지기"인가
- Next.js 공식 인증 가이드(`node_modules/next/dist/docs/01-app/02-guides/authentication.md`)는 **쿠키 세션 + Proxy/Middleware(서버)** 로 라우트를 보호하라고 안내한다.
- **그런데 우리 앱은 토큰을 `localStorage`에 저장한다**(Story 1.3에서 취미 단계 단순화로 결정, 쿠키 전환은 deferred). **`localStorage`는 서버(미들웨어/Proxy)에서 읽을 수 없다.** 따라서 미들웨어 기반 보호는 지금 구조와 맞지 않는다.
- 그래서 이 스토리는 **클라이언트 컴포넌트(AuthGuard)가 마운트 시 토큰을 확인**하고, `GET /api/auth/me`로 유효성까지 검증해 통과/차단하는 SPA 표준 패턴을 쓴다.
- **보안 관점 정직하게:** 화면 쪽 문지기는 "UX 차원의 차단(보호 화면을 안 그림)"이다. **진짜 자물쇠는 백엔드**다 — 환자 데이터 API는 이미 `get_current_user`(security.py)로 토큰을 요구하므로, 미로그인 사용자가 우회로 화면 코드를 봐도 **데이터는 백엔드가 막는다.** [Source: backend/app/security.py#get_current_user]
- Next 공식 문서가 "레이아웃/최상위에서 `return null`로 막는 SPA 패턴은 (다중 진입점·Server Action 때문에) 비권장"이라 적은 것도 같은 취지 — **그래서 우리는 데이터 보호를 백엔드에 두고, 프론트 가드는 보조 수단으로만 쓴다.** 미들웨어/쿠키 전환은 deferred로 남긴다.

### 핵심 코드 가이드 (개발 시 참고)

- **AuthGuard(화면 문지기) 골격:**
  ```tsx
  "use client";
  import { useEffect, useState } from "react";
  import { useRouter } from "next/navigation";
  import { getToken, clearToken } from "@/lib/auth";

  const API_BASE = process.env.NEXT_PUBLIC_API_BASE_URL ?? "http://localhost:8000";

  export function AuthGuard({ children }: { children: React.ReactNode }) {
    const router = useRouter();
    const [status, setStatus] = useState<"checking" | "authed">("checking");

    useEffect(() => {
      const token = getToken();
      if (!token) { router.replace("/login"); return; }
      let cancelled = false;
      (async () => {
        try {
          const res = await fetch(`${API_BASE}/api/auth/me`, {
            headers: { Authorization: `Bearer ${token}` },
          });
          if (cancelled) return;
          if (res.ok) { setStatus("authed"); }
          else { clearToken(); router.replace("/login"); }   // 401 등
        } catch {
          if (!cancelled) { router.replace("/login"); }       // 백엔드 연결 불가 → 안전하게 차단
        }
      })();
      return () => { cancelled = true; };
    }, [router]);

    if (status === "checking") {
      return (
        <main className="flex min-h-screen items-center justify-center bg-bg-base p-8">
          <p className="text-text-secondary">확인 중…</p>
        </main>
      );
    }
    return <>{children}</>;   // 통과한 경우에만 본문 렌더(AC #4)
  }
  ```
  - ⚠️ `useEffect`는 클라이언트에서만 도므로 SSR/프리렌더 시 본문이 새어 나가지 않게 **`checking` 동안 본문을 그리지 않는 것**이 핵심(AC #4).
  - ⚠️ 비동기 응답 도착 전 언마운트 대비 `cancelled` 가드로 "리렌더 경고" 방지.

  ```tsx
      <Button
        variant="outline"
        className="h-11"
        onClick={() => { clearToken(); router.replace("/login"); }}
      >
        로그아웃
      </Button>
    );
  }
  ```

- **홈(`/`) 보호 적용:** 서버 컴포넌트인 `page.tsx`에서 본문을 `<AuthGuard>`로 감싸고 그 안/적절한 위치에 `<LogoutButton />`을 둔다. (AuthGuard·LogoutButton은 클라이언트 컴포넌트라 컴포지션으로 끼워 넣으면 됨.)

### 흔한 실수 방지 (이것만은 피하기)
- ❌ **미들웨어(`middleware.ts`/Proxy)로 막으려 함** → localStorage 토큰은 서버에서 못 읽음. 이번엔 클라이언트 가드로. (쿠키+미들웨어는 deferred)
- ❌ **확인 중에 본문을 먼저 렌더** → 환자 정보가 깜빡 노출. 반드시 `checking` 게이트로 막기(AC #4).
- ❌ **`/login`에 AuthGuard 적용** → 무한 리다이렉트. 로그인 화면은 보호 제외(AC #6).
- ❌ `router.push` 사용 → 뒤로가기로 보호 화면 복귀 가능. **`router.replace`** 사용.
- ❌ `next/router`(구버전) import → App Router는 `next/navigation`.
- ❌ 토큰 유효성 검증을 생략하고 "존재만" 확인 → 만료/위조 토큰이 통과. `/api/auth/me`로 실제 검증(AC #3).
- ❌ 역할별 분기·관리자 화면 만들기 → **Epic 5 범위.** 지금은 "로그인했나/안 했나"만.
