# Story 5.5: 기준값 설정

Status: done

## Story

As a 병원 관리자,
I want "대기 초과 알림 기준 시간" 같은 값을 직접 정하기를,
So that 우리 병원 상황에 맞게 조정한다. (FR14)

## Acceptance Criteria

1. **설정 화면 진입** — 관리자 페이지(`/admin`)의 **"기준값 설정" 카드가 활성 링크**가 되어 `/admin/settings`로 이동한다. 이 화면은 **관리자만**(NFR2) 접근한다.
2. **현재 값 표시** — 설정 화면에 **현재 "대기 초과 기준 시간(분)"** 값이 입력칸에 채워진 채로 보인다(서버에서 읽어옴).
3. **값 변경·저장** — 관리자가 기준 시간을 바꾸고 **저장**하면 그 값이 서버에 저장되고(`PUT /api/admin/settings`), 저장 성공 안내가 보인다.
4. **실제 알림에 적용(FR10 핵심)** — 저장한 기준값이 **대기 초과 판정에 즉시 반영**된다: ① `GET /api/patients`의 `is_overdue`(4.4 대기 초과 알림·흐름판) ② `GET /api/admin/dashboard`의 `overdue_count`·`overdue_threshold_minutes`(5.4 대시보드)가 모두 새 기준값을 쓴다. (하드코딩 상수가 아니라 저장된 설정을 읽는다.)
5. **값 검증** — 기준 시간은 **양의 정수(1~1440분, 즉 1분~24시간)**만 허용. 0·음수·범위 밖·숫자 아님은 **422**로 거부하고 친절한 안내. (화면도 같은 범위로 막되, 진짜 자물쇠는 백엔드.)
6. **관리자 전용 + 보호** — 설정 API(`GET`·`PUT /api/admin/settings`)는 `get_current_admin`으로 보호 — 일반 직원 **403**, 미인증 **401**.
7. **상태 표시** — 로딩 중·오류(친절 안내)·저장 중·저장 완료 상태를 구분해 보여주고, 저장 버튼은 44px·이중 제출 방지(`busyRef`)·저장 중 비활성. 401/403→로그아웃.
8. **기본값·무회귀** — 설정이 아직 없으면 기본 **30분**(`STAGE_OVERDUE_MINUTES`)으로 동작한다(기존 4.4/5.4 동작과 동일). 설정 도입으로 기존 대기 초과 동작이 깨지지 않는다.

> **범위 밖(하지 말 것):**
> - **다른 기준값들** — 이 스토리는 **"대기 초과 기준 시간" 1개**만 다룬다. (`AppSetting` 테이블은 key/value로 일반적으로 만들되, 화면엔 이 값 하나만 노출.) 단계별(접수/진료/검사/수납) 차등 기준·투약 알림 기준 등은 후속.
> - **설정 변경 감사 로그**(누가 언제 바꿨나) — 후속(Epic 5 deferred RBAC/감사 계열).
> - **진짜 푸시 알림·스케줄러** — 범위 밖(기존처럼 화면 열 때 pull 계산).
> - **기존 화면의 추가 게이팅·다른 관리 기능** — 범위 밖.

    @app.get("/api/admin/settings")

        current: Staff = Depends(get_current_admin),
        session: Session = Depends(get_session),

    def update_settings(
        payload: SettingsUpdateIn,
        current: Staff = Depends(get_current_admin),
        session: Session = Depends(get_session),

        v = payload.stage_overdue_minutes
        if v < OVERDUE_MINUTES_MIN or v > OVERDUE_MINUTES_MAX:

                status_code=422,

        session.commit()

            "max": OVERDUE_MINUTES_MAX,

## Dev Notes

### 이 스토리의 핵심 (한 줄)
**하드코딩 기준값을 DB 설정으로.** `STAGE_OVERDUE_MINUTES`(상수)를 `AppSetting`(key/value) 테이블로 옮기고, 관리자가 `/admin/settings`에서 "대기 초과 기준 시간"을 바꿔 저장하면, **4.4 대기 초과 알림(`is_overdue`)과 5.4 대시보드(`overdue_threshold_minutes`)가 그 값을 읽어 즉시 반영**된다(FR14 → FR10 연결). Epic 5의 마지막 스토리.

### ⚠️ 이 스토리의 가장 중요한 설계 판단

1. **기준값은 DB 설정(`AppSetting` key/value), 상수는 "기본값"으로 잔존.**
   - `STAGE_OVERDUE_MINUTES=30`을 **지우지 말 것** — `get_overdue_threshold`의 폴백 + 시드 기본값으로 계속 쓴다. 설정 행이 없거나 깨져도 30분으로 안전 동작(무회귀, AC8).
   - key/value 일반 저장소로 만들어 향후 다른 기준값도 같은 테이블 재사용(단, 이번 화면은 1개만).

2. **읽기 경로는 "파라미터 주입"으로(전역 가변 상태 금지).**
   - `stage_wait_info(entered_at, now, threshold=STAGE_OVERDUE_MINUTES)`로 기본값 파라미터를 추가 → **기존 호출 하위호환**, 테스트·추론 쉬움. 모듈 전역을 런타임에 바꾸는 방식(가변 글로벌)은 피한다(워커별 불일치·경쟁).
   - 호출부 2곳(`list_patients`·`admin_dashboard`)만 요청당 한 번 `get_overdue_threshold(session)`를 읽어 주입. (요청마다 1회 작은 조회 — 캐시는 과한 최적화, 범위 밖.)

3. **반영은 "읽을 때 계산"으로 자동.** 4.4·5.4가 이미 `stage_wait_info`/`overdue_threshold_minutes`를 통해 매 요청 계산하므로, 저장만 하면 다음 조회부터 새 값이 적용된다(별도 브로드캐스트·캐시 무효화 불필요). 진짜 푸시·스케줄러는 범위 밖(기존 pull 방식 유지).

4. **진짜 자물쇠는 백엔드.** `get_current_admin`(403)으로 GET/PUT 보호 + 범위 검증(422). 화면 입력 제한·AdminGuard는 UX 보조(5.1/5.4 철학). RSC 프리렌더 금지(1.4).

5. **PUT 업서트 + 멱등 시드.** 단일 값이라 PUT(전체 교체). 행이 없으면 생성·있으면 갱신. lifespan 시드는 admin1 블록과 동일한 "키 존재 확인 후 생성"(멱등). 멀티워커 시드 레이스는 기존 deferred 계열.
