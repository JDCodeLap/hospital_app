# Story 5.2: 직원 계정 관리

Status: review

## Story

As a 병원 관리자,
I want 직원 계정을 등록·수정·삭제하고 직군(의사/간호사/원무과 등)을 지정하기를,
So that 누가 앱을 쓰는지 관리한다. (FR11)

## Acceptance Criteria

1. **직원 목록 조회** — 관리자 페이지에서 전체 직원 목록을 본다. 각 직원의 **아이디·이름·권한(관리자/일반직원)·직군·활성 여부**가 보인다. **비밀번호(해시 포함)는 절대 화면·응답에 나오지 않는다**(NFR3).
2. **직원 등록** — 관리자가 새 직원을 추가한다(아이디·비밀번호·이름·권한·직군). 저장하면 목록에 즉시 반영된다. **아이디가 이미 있으면** "이미 사용 중인 아이디입니다" 안내(중복 거부).
3. **직원 수정** — 관리자가 기존 직원의 **이름·권한·직군·활성 여부**를 바꾼다. **비밀번호는 새로 입력했을 때만** 변경되고, 비워두면 기존 비밀번호가 유지된다. (아이디는 변경하지 않는다 — 로그인 식별자 안정성)
4. **직원 삭제** — 관리자가 직원을 삭제하면 목록에서 사라진다. 삭제 전 한 번 더 확인을 요구한다.
5. **직군 지정** — 각 직원에 **직군(의사/간호사/원무과/기타)**을 지정할 수 있다. 직군은 **권한(role)과 별개**다 — 직군은 "어떤 일을 하는 사람인지"(표시·분류), 권한은 "관리자 페이지 접근 가능 여부"(admin/staff). 직군은 정보 접근 범위를 바꾸지 않는다(범위 권한은 Story 5.3).
6. **자기 잠금(self-lockout) 방지** — 시스템에 **관리자가 0명이 되는 일을 막는다**: 마지막 남은 관리자의 권한을 일반직원으로 강등하거나, 삭제하거나, 비활성화하려 하면 **거부**하고 안내한다. 또한 관리자는 **로그인 중인 자기 자신을 삭제할 수 없다**.
7. **관리자 전용 + 보호** — 직원 관리 기능(화면·API)은 **관리자만**(NFR2) 접근한다. 직원 관리 API(`/api/admin/staff*`)는 일반 직원 호출 시 **403**, 미인증 시 **401**을 반환한다(5.1 `get_current_admin` 재사용).
8. **권한 표기 정규화** — 권한 값은 저장 시점에 정규화(`strip().lower()`)되어 **`admin` 또는 `staff`만** 허용된다(그 외 값은 거부). 정규화 덕분에 `" Admin"` 같은 입력도 메뉴 노출(AppShell)·관리자 게이트(get_current_admin)와 정확히 일치한다(5.1에서 미뤄둔 항목 이행).

> **범위 밖(하지 말 것):**
> - **직원별 정보 접근 범위(볼 수 있는 정보) 설정 = Story 5.3(FR12)** — 이번 `role`은 여전히 **admin/staff 2단계**(관리자 페이지 접근 여부)일 뿐. 직군(job_title)은 분류용이며 **어떤 화면/환자도 제한하지 않는다**(일반 직원은 지금까지처럼 모든 환자 기능 사용).
> - **전체 현황 대시보드(오늘 환자 수·과별 혼잡도·평균 대기시간) = Story 5.4(FR13)**.
> - **기준값 설정(대기 초과 시간 등) = Story 5.5(FR14)** — `STAGE_OVERDUE_MINUTES`는 계속 하드코딩.
> - **접근 감사 로그**(누가 어떤 직원을 추가/삭제했는지 기록) — 후속. 이번 범위 아님.
> - **비밀번호 정책(복잡도·만료)·이메일 인증·초대 링크·직원 본인 셀프 프로필 수정** — 범위 밖(관리자만 CRUD).
> - **Alembic 마이그레이션 도입** — 이번엔 `Staff.job_title` 한 컬럼만 멱등 DDL로 추가(아래 설계 판단 #2). 풀 마이그레이션 체계는 계속 deferred.

    ```python
    # 권한(role)은 2단계만(관리자 페이지 접근 여부). 세분화 권한은 Story 5.3.
    VALID_ROLES = {"admin", "staff"}
    # 직군(job_title)은 분류·표시용(권한과 무관). 자유도를 위해 "기타"도 둔다.
    JOB_TITLES = ["의사", "간호사", "원무과", "기타"]

    def normalize_role(raw: str) -> str:
        """권한 표기를 정규화. 'admin'/'staff'만 허용, 그 외는 422.
        5.1 deferred 이행: ' Admin' 같은 입력도 메뉴/게이트와 일치하도록 strip().lower()."""
        role = (raw or "").strip().lower()
        if role not in VALID_ROLES:
            raise HTTPException(status_code=422, detail="권한은 admin 또는 staff만 가능합니다")
        return role
    ```

## Dev Notes

### 이 스토리의 핵심 (한 줄)
**관리자 영역의 첫 실제 기능 — 직원 CRUD.** 5.1이 만든 `/admin`·`get_current_admin` 위에 `/api/admin/staff*`(목록·등록·수정·삭제) + `/admin/staff` 화면을 얹는다. **권한(role)과 직군(job_title)을 분리**하고, **마지막 관리자 보호**·**권한 정규화**로 안전하게.

### ⚠️ 이 스토리의 가장 중요한 설계 판단

1. **권한(role)과 직군(job_title)은 다르다 — 섞지 마라.** (5.1 인계의 핵심 지시)
   - `role` = `admin` / `staff` **2단계**. "관리자 페이지에 들어갈 수 있는가". `get_current_admin`이 이걸로 게이트. **세분화(정보 접근 범위)는 5.3**.
   - `job_title` = `의사`/`간호사`/`원무과`/`기타`. "어떤 일을 하는 사람인가"(표시·분류). **어떤 화면도 제한하지 않는다.** 직군을 role에 섞으면(예: role="의사") 게이트가 깨지고 5.3 권한 설계가 꼬인다.
   - 그래서 **`Staff`에 `job_title` 컬럼을 따로 추가**한다(role 재사용 안 함).

2. **첫 "기존 테이블 변경" — 멱등 DDL로 우회(Alembic 아직 도입 안 함).**
   - 그동안(SafetyAck·MedicationAdministration·ChecklistCheck·HandoverNote·StageEntry)은 **새 테이블**로 만들어 `create_all`만으로 끝냈다(create_all은 없는 테이블만 만들고 기존 테이블 ALTER는 못 함 — 1-2 deferred).
   - `job_title`은 Staff의 1:1 속성이라 **새 테이블 분리가 부자연**스럽다 → `Staff`에 컬럼 추가 + `lifespan`에서 **`ALTER TABLE staff ADD COLUMN IF NOT EXISTS job_title ...`**(Postgres 멱등 DDL). 기존 데이터 보존, 재시작 안전, 빈 DB는 create_all이 컬럼 포함 생성.
   - **Alembic 풀 도입은 계속 deferred**(데모 규모엔 과함). 단, 이 케이스를 deferred-work에 "기존 테이블 변경을 멱등 DDL로 처리 — 스키마가 더 커지면 Alembic 필요"로 환기.

3. **마지막 관리자 보호(self-lockout 방지)가 이 스토리의 핵심 안전장치.**
   - 관리자가 **마지막 admin을 강등/삭제/비활성화**하면 시스템에 관리자가 0명 → 누구도 `/admin`에 못 들어가는 영구 잠금. 백엔드에서 **활성 admin 수 ≤ 1이면 거부(409)**.
   - **자기 자신 삭제 금지**도 추가(실수 방지). 자기 강등은 마지막 admin 가드로 충분히 커버(자기가 마지막이면 막힘, 아니면 허용하되 즉시 자기 메뉴가 사라지는 건 정상).
   - **프론트 버튼 비활성에만 의존하지 않고 백엔드가 강제**(3.1/5.1 철학). 진짜 자물쇠는 서버.

4. **권한 정규화(5.1 deferred 이행).** 쓰기(POST/PATCH) 시점에 `role.strip().lower()` 후 `{admin, staff}`만 허용(아니면 422). 5.1까지는 role을 만드는 경로가 시드뿐이라 무해했지만, **이제 관리자가 role을 편집**하므로 `" Admin"`·`"ADMIN"` 같은 표기가 조용히 권한 미부여(메뉴·게이트 불일치)를 일으킬 수 있다. 정규화로 프론트(`=== "admin"`)·백엔드(`!= "admin"`)·DB 값이 항상 일치.

5. **비밀번호: 해시 저장·미노출, 수정 시 빈 값이면 유지.** `hash_password`(1.3) 재사용. **응답에 `hashed_password`를 절대 넣지 않는다**(`staff_public` 헬퍼로 직렬화 단일화 — 누락 실수 방지). 수정 폼에서 비번을 비워 보내면 기존 해시를 건드리지 않는다(흔한 실수: 빈 문자열을 해시해 덮어쓰면 로그인 불가).

6. **삭제는 하드 삭제 + FK 친절 처리 → 비활성화 경로 제공.** 직원이 남긴 기록(SafetyAck·MedicationAdministration·ChecklistCheck·HandoverNote의 FK)이 있으면 하드 삭제가 IntegrityError(500) → 잡아서 **409 + "비활성화하세요"** 안내. `Staff.is_active`는 이미 있고 `authenticate_token`이 비활성 직원의 토큰을 거부하므로(security.py:69), **비활성화 = 사실상 로그인 차단**(soft delete). 기록 없는 신규 직원은 하드 삭제 성공.

7. **모든 직원 관리 API는 `get_current_admin` 보호(5.1 재사용).** 일반 직원 403·미인증 401. 화면(AdminGuard)·메뉴는 보조. RSC 프리렌더 금지(1.4) — AdminGuard 통과 후 클라이언트 fetch.

```python
# main.py — 상수/헬퍼 (CHECKLIST_ITEMS 등 상수 영역 옆에)
VALID_ROLES = {"admin", "staff"}
JOB_TITLES = ["의사", "간호사", "원무과", "기타"]

def normalize_role(raw: str) -> str:
    role = (raw or "").strip().lower()
    if role not in VALID_ROLES:
        raise HTTPException(status_code=422, detail="권한은 admin 또는 staff만 가능합니다")
    return role

def staff_public(s: Staff) -> dict[str, object]:
    # 비밀번호 해시는 절대 포함하지 않는다(단일 직렬화 지점)
    return {"id": s.id, "username": s.username, "full_name": s.full_name,
            "role": s.role, "job_title": s.job_title, "is_active": s.is_active}

def assert_not_last_admin(session: Session, target: Staff) -> None:
    if target.role == "admin" and target.is_active:
        active_admins = session.exec(
            select(Staff).where(Staff.role == "admin", Staff.is_active == True)  # noqa: E712
        ).all()
        if len(active_admins) <= 1:
            raise HTTPException(status_code=409,
                detail="마지막 관리자는 강등·삭제·비활성화할 수 없습니다")
```

```python
# main.py — 엔드포인트 (전부 Depends(get_current_admin) 보호)
@app.get("/api/admin/staff")
def list_staff(current: Staff = Depends(get_current_admin),
               session: Session = Depends(get_session)) -> list[dict[str, object]]:
    rows = session.exec(select(Staff).order_by(Staff.username)).all()
    return [staff_public(s) for s in rows]

@app.post("/api/admin/staff")
def create_staff(payload: StaffCreateIn,
                 current: Staff = Depends(get_current_admin),
                 session: Session = Depends(get_session)) -> dict[str, object]:
    username = payload.username.strip()
    if not username:
        raise HTTPException(status_code=422, detail="아이디를 입력하세요")
    if not payload.password.strip():
        raise HTTPException(status_code=422, detail="비밀번호를 입력하세요")
    role = normalize_role(payload.role)
    # 사전 중복 확인(친절한 메시지) + IntegrityError 폴백(경합 안전)
    if session.exec(select(Staff).where(Staff.username == username)).first():
        raise HTTPException(status_code=409, detail="이미 사용 중인 아이디입니다")
    staff = Staff(username=username, hashed_password=hash_password(payload.password),
                  full_name=payload.full_name.strip(), role=role,
                  job_title=payload.job_title.strip(), is_active=True)
    session.add(staff)
    try:
        session.commit()
    except IntegrityError:
        session.rollback()
        raise HTTPException(status_code=409, detail="이미 사용 중인 아이디입니다")
    session.refresh(staff)
    return staff_public(staff)

@app.patch("/api/admin/staff/{staff_id}")
def update_staff(staff_id: int, payload: StaffUpdateIn,
                 current: Staff = Depends(get_current_admin),
                 session: Session = Depends(get_session)) -> dict[str, object]:
    staff = session.get(Staff, staff_id)
    if staff is None:
        raise HTTPException(status_code=404, detail="직원을 찾을 수 없습니다")
    # 마지막 admin 보호: '관리자에서 빠지는' 변경이면 검사
    new_role = normalize_role(payload.role) if payload.role is not None else staff.role
    demoting = staff.role == "admin" and new_role != "admin"
    deactivating = payload.is_active is False and staff.is_active
    if demoting or deactivating:
        assert_not_last_admin(session, staff)
    if payload.full_name is not None:
        staff.full_name = payload.full_name.strip()
    if payload.job_title is not None:
        staff.job_title = payload.job_title.strip()
    if payload.role is not None:
        staff.role = new_role
    if payload.is_active is not None:
        staff.is_active = payload.is_active
    if payload.password is not None and payload.password.strip():
        staff.hashed_password = hash_password(payload.password)
    session.add(staff)
    session.commit()
    session.refresh(staff)
    return staff_public(staff)

@app.delete("/api/admin/staff/{staff_id}")
def delete_staff(staff_id: int,
                 current: Staff = Depends(get_current_admin),
                 session: Session = Depends(get_session)) -> dict[str, object]:
    staff = session.get(Staff, staff_id)
    if staff is None:
        raise HTTPException(status_code=404, detail="직원을 찾을 수 없습니다")
    if staff.id == current.id:
        raise HTTPException(status_code=409, detail="자기 계정은 삭제할 수 없습니다")
    assert_not_last_admin(session, staff)  # 대상이 마지막 admin이면 막음
    session.delete(staff)
    try:
        session.commit()
    except IntegrityError:
        # 이 직원이 남긴 기록(FK)이 있어 하드 삭제 불가 → 비활성화 안내
        session.rollback()
        raise HTTPException(status_code=409,
            detail="이 직원은 활동 기록이 있어 삭제할 수 없습니다. 대신 비활성화하세요.")
    return {"deleted": staff_id}
```

- ❌ 직군을 `role`에 욱여넣기(role="의사") → 게이트(admin/staff) 붕괴·5.3 권한설계 오염. **`job_title` 별도 컬럼**으로.
- ❌ `create_all`만 믿고 `job_title` 컬럼이 기존 테이블에 생길 거라 가정 → ALTER 안 됨. **멱등 `ADD COLUMN IF NOT EXISTS` 필수**.
- ❌ 응답에 `Staff`를 통째로 반환(`hashed_password` 노출) → 반드시 `staff_public`로 직렬화.
- ❌ 수정 시 빈 비밀번호를 해시해 덮어쓰기 → 로그인 불가. **빈/None이면 비번 미변경**.
- ❌ 마지막 관리자 강등/삭제/비활성화 허용 → 영구 잠금. **`assert_not_last_admin` 가드 필수**(서버에서).
- ❌ role 정규화 누락 → `" Admin"`이 staff처럼 취급돼 메뉴/게이트 불일치. **쓰기 시 `normalize_role`**.
- ❌ 하드 삭제만 시도하고 FK IntegrityError를 500으로 흘림 → **잡아서 409 친절 안내(비활성화 경로)**.
- ❌ 자기 자신 삭제 허용 → 실수로 자기 계정 날림. **`staff_id == current.id` 차단**.
- ❌ 프론트 비활성/숨김에만 의존 → 진짜 자물쇠는 백엔드 `get_current_admin` + 가드(409). 화면은 보조.
- ❌ `/admin/staff` 서버 컴포넌트에서 직원 데이터 프리렌더 → 1.4 금지. 클라이언트(StaffManagement)에서 토큰 달고 호출.
- ❌ AppShell NAV에 직원관리 항목 추가 → 메뉴 비대화. `/admin` 랜딩 카드/URL로 진입.
- ❌ `/api/auth/me`에 직군 끼워넣어 화면 확장 → 불필요(화면은 `/api/admin/staff` 사용). 변경 최소화.
