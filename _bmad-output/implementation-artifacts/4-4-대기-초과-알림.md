# Story 4.4: 대기 초과 알림

Status: done

## Story

As a 병원 직원,
I want 한 환자가 한 단계(접수/진료/검사/수납)에 너무 오래 멈춰 있으면 흐름판에서 눈에 띄게 강조되고 알림 화면에도 뜨기를,
So that 방치되는 환자 없이 제때 다음 처리를 진행한다. (FR10)

## Acceptance Criteria

1. 환자가 **현재 단계에 진입한 시각**이 서버에 기록된다 — 단계가 바뀔 때(3.4 `advance-stage`) 그 시각이 갱신된다.
2. 서버가 **현재 시각 − 단계 진입 시각**으로 대기 시간을 계산하고, **기준 시간(기본 30분)** 이상이면 그 환자를 **대기 초과(overdue)**로 판정한다.
3. **흐름판(`/board`, 4.3)** 에서 대기 초과 환자 카드가 **앰버 색으로 강조**된다 — 색만이 아니라 **아이콘+글자**("대기 초과" 또는 "N분 대기")를 함께 표시(UX-DR8).
4. **알림 화면(`/alerts`, 3.3)** 에 **"대기 초과 환자" 섹션**이 투약 알림과 함께 표시된다 — 대기 초과 환자 목록(이름·현재 단계·대기 시간), 각 항목은 통합 화면(`/patients/{id}`)으로 링크. 대기 초과가 없으면 그 섹션은 "없음"/숨김. (= 담당자 알림 surface)
5. **백엔드 `GET /api/patients`** 응답 각 항목에 `waiting_minutes`(정수, 진입기록 없으면 null)와 `is_overdue`(bool)가 **추가**된다. 기존 키(`id·registration_number·name·age·gender·current_stage·allergies`)는 전부 유지(하위호환).
6. 기준 시간은 **백엔드 상수**(`STAGE_OVERDUE_MINUTES = 30`)로 둔다 — 관리자 설정(5.5/FR14)은 범위 밖, 지금은 하드코딩.
7. **로그인한 직원만** 조회한다(NFR3). 기존 흐름판·알림·검색·통합화면·WS 동작을 깨지 않는다.
8. **새 테이블로 마이그레이션 0** — `StageEntry`는 새 테이블이라 `create_all`로 자동 생성(Patient 컬럼 추가는 `create_all`이 ALTER 못 하므로 금지). 기존 8명 환자는 **서버 시작 시 백필**로 진입 기록을 채운다(데모로 일부가 대기 초과로 보이도록 시각 분산).

> **범위 밖(하지 말 것):**
> - **기준 시간 관리자 설정 UI = Story 5.5(FR14)** — 지금은 상수 하드코딩.
> - **진짜 푸시 알림(휴대폰 푸시/이메일/SMS)** — "알림"은 3.3과 동일하게 화면(/alerts)에 모아 보여주는 방식. 예약 푸시·스케줄러는 범위 밖(3.3과 동일 철학).
> - **단계별로 다른 기준 시간**(접수 10분/검사 60분 등) — 지금은 전 단계 단일 기준. 단계별 차등은 후속/5.5.
> - **대기 초과 이력·통계·평균 대기시간**(관리자 대시보드) = Epic 5(FR13).
> - **단계 진입 이력 로그(append-only history)** — `StageEntry`는 **환자당 1건(현재 단계 진입 시각만)**. 전체 이력 추적은 후속.
> - **흐름판 전역 실시간 WebSocket** — 4.3과 동일(로드+새로고침, 60초 자동갱신은 /alerts에 한해 기존 패턴 유지).

    ```python
    def backfill_stage_entries(session: Session) -> None:
        now = datetime.now()
        existing = {e.patient_id for e in session.exec(select(StageEntry)).all()}
        patients = session.exec(select(Patient).order_by(Patient.id)).all()
        added = False
        for i, p in enumerate(patients):
            if p.id in existing:
                continue
            offset = _BACKFILL_OFFSETS_MIN[i % len(_BACKFILL_OFFSETS_MIN)]
            session.add(StageEntry(patient_id=p.id, stage=p.current_stage,
                                   entered_at=now - timedelta(minutes=offset)))
            added = True
        if added:
            session.commit()
    ```

## Dev Notes

### 이 스토리의 핵심 (한 줄)
새 테이블 `StageEntry`(환자당 현재 단계 진입 시각)로 **대기 시간**을 계산해, 기준(30분) 초과 환자를 **흐름판 앰버 강조 + /alerts 대기초과 섹션**으로 알린다. 진입 시각은 3.4 advance-stage가 갱신, 기존 환자는 시작 시 백필.

### ⚠️ 이 스토리의 가장 중요한 설계 판단

1. **`stage_entered_at`은 Patient 컬럼이 아니라 새 테이블 `StageEntry`.** `Patient`에 컬럼을 추가하면 `SQLModel.metadata.create_all`이 **기존 테이블을 ALTER하지 못해** 컬럼이 안 생긴다(1-2 deferred 이슈, Alembic 미도입). 그래서 3.2 SafetyAck·3.3 MedicationAdministration·3.4 ChecklistCheck·4.2 HandoverNote와 **동일하게 새 테이블**로 우회 → `create_all`이 자동 생성, 마이그레이션 0. **환자당 1건(unique patient_id), upsert**(이력 로그 아님).

2. **진입 시각 갱신 지점은 단 하나: `advance-stage`(3.4).** 단계 변경의 유일 경로가 advance-stage이므로(4.1/4.3에서 확인) 거기서만 StageEntry를 갱신하면 항상 정확하다. 다른 곳에서 current_stage를 바꾸지 않는다.

3. **기존 8명 환자 백필 필요.** 진입 시각 기록은 이 기능 이후 advance한 환자에게만 생긴다. 이미 시드된 8명은 기록이 없으므로 **서버 시작 시 백필**로 채운다. **데모 가시성**을 위해 진입 시각을 분산(`_BACKFILL_OFFSETS_MIN`)해 30분 기준으로 여러 명이 즉시 "대기 초과"로 보이게 한다(앱의 "그럴듯한 시드 데이터" 관행과 일치 — 8명 환자도 단계·알레르기를 다양하게 시드함). **멱등**: 이미 기록 있는 환자는 건너뜀 → 재시작해도 시각 고정.
   - ※ 백필 시각은 한 번 정해지면 고정이라, 실시간이 흐르면 결국 더 많은 환자가 초과로 보일 수 있음(데모 특성). advance-stage하면 그 환자는 now로 리셋되어 정상으로 돌아온다.

4. **읽기 surface는 새 엔드포인트 없이 확장.** 흐름판도 /alerts 대기초과 섹션도 **기존 `GET /api/patients`**(4.3에서 전체 호출)에 `waiting_minutes`·`is_overdue`를 추가해 재사용한다. 새 `/api/overdue-patients`는 만들지 않음(중복 회피) — /alerts 섹션은 GET /api/patients를 호출해 `is_overdue`로 필터.

5. **앰버 강조는 opt-in prop.** `PatientCard`에 `overdue?` prop 추가(4.3 `hideStage`와 동일 패턴). 흐름판만 `overdue={p.is_overdue}` 전달 → **검색 화면은 시각 변화 없음**. 색만이 아니라 아이콘+"대기 초과" 글자 병행(UX-DR8).

6. **기준 시간은 상수.** `STAGE_OVERDUE_MINUTES = 30` 하드코딩. 관리자가 바꾸는 것(FR14)은 5.5. 단계별 차등도 후속.

7. **"알림"=화면 모아보기**(3.3 철학). 진짜 푸시 아님. /alerts에 섹션 추가가 "담당자에게 알림이 간다"의 구현.

### 흔한 실수 방지

- ❌ `Patient`에 `stage_entered_at` **컬럼 추가** → `create_all`이 ALTER 못 해 안 생김. **새 테이블 `StageEntry`** 사용.
- ❌ StageEntry를 **이력 로그(append-only)로** 만들기 → 이번은 환자당 1건 upsert(현재 진입 시각만). 이력은 범위 밖.
- ❌ 백필을 **비멱등**으로 → 재시작마다 시각이 바뀌어 대기 초과가 리셋. 반드시 "기록 없는 환자만" 생성.
- ❌ advance-stage에서 StageEntry 갱신 빠뜨림 → 단계는 바뀌는데 대기 시계가 안 리셋되어 영원히 초과로 보임.
- ❌ `GET /api/patients`에 `is_overdue` 추가하며 기존 키 제거 → **기존 키 전부 유지**(하위호환, 검색/흐름판 깨짐 방지).
- ❌ StageEntry를 환자마다 개별 쿼리(N+1) → 한 번에 조회해 맵으로. `now`도 루프 밖 1회.
- ❌ `MedicationAlerts.tsx`를 수정 → 별도 `OverduePatients.tsx` 신설(3.3 회귀 방지).
- ❌ 앰버를 검색 화면까지 적용 → `overdue` prop 흐름판만 전달(기본 false).
- ❌ 색만으로 강조 → 아이콘+"대기 초과/N분" 글자 병행(UX-DR8).
- ❌ `python -c`에 한글 → UTF-8 파일 스크립트로(4.3에서 분포 검증 깨졌던 사례).
