# Story 4.3: 환자 흐름판 (전체 현황)

Status: done

## Story

As a 병원 직원,
I want 모든 환자가 지금 어느 단계(접수→진료→검사→수납)인지 한 화면에서 보기를,
So that 현장 전체 상황을 한눈에 파악한다. (FR9)

## Acceptance Criteria

1. 새 화면 `/board`(환자 흐름판)가 있다 — 로그인한 직원만 접근(AuthGuard 보호, 미인증은 `/login`).
2. 흐름판을 열면 **모든 환자**가 **현재 단계별로 묶여(보드/그룹 목록)** 표시된다 — 4개 단계(접수·진료·검사·수납) 각각에 그 단계에 있는 환자 카드들이 모인다.
3. 각 단계 그룹은 **단계 이름 + 그 단계의 환자 수**를 보여준다(예: "진료 · 3명"). 환자가 없는 단계는 "없음"(또는 0명)으로 표시.
4. 각 환자 카드는 **이름 · 나이 · 등록번호**를 보여주고, 알레르기가 있으면 위험 배지(아이콘+글자)를 함께 보여준다(2.2 카드 패턴 재사용, 색만으로 정보 전달 금지=UX-DR8).
5. 환자 카드를 누르면 그 환자의 **통합 화면(`/patients/{id}`)**으로 이동한다.
6. **새로고침 버튼**이 있어 누르면 최신 상태로 다시 불러온다. 로딩 중·오류·환자 없음(전체 0명) 상태를 각각 안내한다.
7. **홈/네비게이션에서 흐름판으로 가는 진입점**이 있다(AppShell `NAV`에 "흐름판" 추가 → 사이드 메뉴·폰 하단 탭에 노출 + 홈 빠른 진입 카드).
8. **백엔드 변경 없음** — 기존 `GET /api/patients`(2.2, 보호됨)가 이미 각 환자의 `current_stage`를 주므로 그대로 재사용한다. 환자 데이터는 Server Component에서 프리렌더하지 않고 통과 후 클라이언트에서 토큰 달아 호출한다(1.4 RSC 가이드).

> **범위 밖(하지 말 것):**
> - **대기 초과 앰버 강조 + 담당자 알림 = Story 4.4(FR10)** — 이번엔 단계 머문 시간 계산·앰버 깜빡임·알림 전송을 **구현하지 않는다**. 흐름판은 "현재 단계 표시"까지만.
> - **단계 변경 UI** — 단계 진행은 3.4 `advance-stage`(통합 화면의 ProcedureChecklist "다음 단계로")가 유일 경로. 흐름판은 **읽기 전용**.
> - **흐름판 전체 실시간(WebSocket) 자동 갱신** — 기존 WS는 환자별 채널(`/ws/patients/{id}`)이라 보드 전역 채널이 없음. 새 전역 채널은 후속. 이번엔 로드 + 새로고침 버튼.
> - **관리자 통계**(오늘 환자 수·과별 혼잡도·평균 대기시간) = Epic 5(FR13).
> - **부서/담당자별 필터·정렬** = Epic 5(권한과 함께).

## Dev Notes

### 이 스토리의 핵심 (한 줄)
**프론트 전용.** 기존 `GET /api/patients`(이미 `current_stage` 포함)를 전체 호출해 **단계별로 묶어 보여주는** 새 화면 `/board` + 컴포넌트 `PatientFlowBoard` + 네비 진입점. 백엔드·새 API 0.

### ⚠️ 이 스토리의 가장 중요한 설계 판단

1. **백엔드 변경 0 — 기존 목록 API 재사용.** `GET /api/patients`는 이미 모든 환자의 `id·name·age·gender·current_stage·allergies·registration_number`를 반환한다(2.2에서 확장 완료). 흐름판이 필요한 데이터가 전부 들어 있으므로 **새 엔드포인트·모델·쿼리 추가 없음**. 그룹화는 클라이언트에서.

2. **흐름판은 읽기 전용.** 단계 변경은 3.4 `advance-stage`(통합 화면)만이 유일 경로. 흐름판에서 드래그·버튼으로 단계를 바꾸지 않는다(범위 밖). 카드는 통합 화면으로 가는 링크만.

3. **STAGE_ORDER 프론트도 동일 정의.** `["접수","진료","검사","수납"]` — StageTimeline.tsx와 백엔드 `STAGE_ORDER`와 같은 순서. 알 수 없는 단계 값은 "기타" 그룹으로 방어(서버가 비표준 단계를 줘도 화면이 안 깨짐).

4. **RSC 프리렌더 금지(1.4 가이드).** 환자 데이터(PHI)는 Server Component에서 미리 렌더하지 않는다 → `board/page.tsx`는 AuthGuard·AppShell·헤더만 두고, 실제 fetch는 클라이언트 `PatientFlowBoard`가 토큰 달고 수행. `/patients`·`/alerts`와 동일 정책(검증에서 초기 HTML에 PHI 미노출 확인).

5. **대기 초과(앰버·알림)는 4.4.** 이번 AC는 "현재 단계 표시"까지만. 단계에 머문 시간을 계산하거나 앰버로 강조하거나 알림을 보내는 로직은 **넣지 않는다**. (Flow 2의 "앰버로 깜빡 + 알림"은 FR10=4.4 몫)

6. **실시간은 이번 범위 밖.** 기존 WebSocket은 `/ws/patients/{id}` 환자별 채널이라 보드 전역 갱신엔 부적합. 보드 전역 WS 채널은 후속. 이번엔 **로드 + 새로고침 버튼**(필요 시 60초 자동 갱신은 `/alerts` 패턴 참고 가능하나 선택).

### 확정 기술/패턴 가이드 (골격)

```tsx
// frontend/src/components/PatientFlowBoard.tsx (NEW, 골격)
"use client";
import { useCallback, useEffect, useState } from "react";
import { useRouter } from "next/navigation";
import { API_BASE } from "@/lib/api";
import { authHeader, clearToken } from "@/lib/auth";
import { Icon } from "@/components/Icon";
import type { PatientSummary } from "@/components/PatientSearch";

const STAGE_ORDER = ["접수", "진료", "검사", "수납"] as const;

export function PatientFlowBoard() {
  const router = useRouter();
  const [patients, setPatients] = useState<PatientSummary[] | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  const load = useCallback(async (signal?: AbortSignal) => {
    setLoading(true);
    try {
      const res = await fetch(`${API_BASE}/api/patients`, { headers: { ...authHeader() }, signal });
      if (signal?.aborted) return;
      if (res.status === 401 || res.status === 403) { clearToken(); router.replace("/login"); return; }
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      const data: PatientSummary[] = await res.json();
      if (signal?.aborted) return;
      setPatients(data); setError(null);
    } catch (err: unknown) {
      if (signal?.aborted) return;
      if (err instanceof DOMException && err.name === "AbortError") return;
      setError("환자 목록을 불러올 수 없습니다. 백엔드 서버가 켜져 있는지 확인하세요.");
      setPatients(null);
    } finally {
      // abort된 경우 로딩 상태 건드리지 않음(다음 요청이 이어받음)
      setLoading(false);
    }
  }, [router]);

  useEffect(() => {
    const c = new AbortController();
    void load(c.signal);
    return () => c.abort();
  }, [load]);

  // STAGE_ORDER로 그룹핑 + 알 수 없는 단계는 "기타"
  // 헤더에 새로고침 버튼(onClick={() => load()}), 단계별 grid 섹션, 카드 = PatientSearch의 PatientCard 재사용
  // ...
}
```

> ⚠️ `lint set-state-in-effect` 주의: 이전 스토리들에서 `setState`를 effect 본문 직접 호출 시 lint 경고가 났다. 위처럼 **비동기 콜백 안에서 setState**(load 함수 내부)하거나 `void load(...)`로 감싸면 안전. PatientDetail의 `void (async()=>{...})()` 패턴 참고.

> ⚠️ `finally`의 `setLoading(false)`: PatientSearch는 `if (!cancelled) setLoading(false)`로 가드한다. 단일 로드(디바운스 없음)라 단순화 가능하나, abort 시 늦은 응답이 로딩을 끄지 않게 하려면 `cancelled` 표식 패턴을 따르는 게 안전(2.2 참고).
